# Задание финала Робошколы 2026

## «Квадропеды: от локомоции к навигации»

### 1. Общая концепция

Задача состоит в том, чтобы разработать и интегрировать программные модули, обеспечивающие автономную навигацию четырёхногого робота с использованием бортовой RGB‑D камеры.

Организаторы предоставляют:
- Базовую RL‑политику для устойчивого перемещения робота (ходьба с управлением по целевым координатам);
- Сконфигурированное программное окружение (симулятор, инструменты разработки, интерфейсы управления).

Участники должны дополнить систему компонентами, которые позволят роботу:
- самостоятельно исследовать помещение;
- обнаруживать заданные объекты по данным камеры;
- планировать маршрут к объектам в заданной последовательности;
- точно подходить к каждому объекту в автономном режиме.

Итоговое тестирование проходит в два этапа:
- Этап 1 – для всех команд тестирование работы в симуляторе
- Этап 2 - топ‑5 команд по результатам этапа 1 тестирование на реальном роботе

### 2. Состав команд

- Размер команды: 3–4 человека, состав определяется жеребьёвкой.
- Время работы: 2 полных дня в очном формате, перед началом работы будет проведён установочный семинар с разбором условий задания и ответом на вопросы.

### 3. Предоставляемые материалы и стек

#### 3.1. Базовое программное обеспечение

- Симулятор (Isaac Gym с моделью четырёхногого робота и датчиками).
- Данные сенсоров, доступные в симуляторе:
  - Положения сочленений, скорости, данные IMU (опционально);
  - Изображения с RGB‑D камеры: два канала — IMAGE_COLOR и IMAGE_DEPTH.
- Базовая стратегия ходьбы: обученный RL‑агент (вход: целевое смещение/скорость; выход: команды суставам). Участники не изменяют низкоуровневый контроллер, но могут вызывать его с различными целевыми значениями.
- Сконфигурированное программное окружение в контейнере. В составе окружения предоставляется ROS2 bridge с предопределёнными топиками для обмена данными, при этом использование ROS является опциональным.

#### 3.2. Данные о сцене и объектах

- Объекты для поиска представляют собой изображения (например, напечатанные маркеры или картинки), которые система компьютерного зрения должна уметь обнаруживать и идентифицировать. Каждому объекту соответствует уникальный идентификатор.
- Примеры объектов в симуляторе и на реальном полигоне (габариты, внешний вид) предоставляются до начала финала.
- Карта помещения не предоставляется — робот должен действовать в неизвестной среде.
- Важно: для целей тестирования и отладки в процессе разработки предоставляется среда с фиксированными конфигурацией пространства и позициями объекта для поиска, при это при валидации решений организаторы будут использовать серию тестов со случайно задаваемыми топологией окружения и расстановкой объектов в нем.

### 4. Задачи участников

#### 4.1. Компонент детекции объектов

- Используя поток RGB‑D изображений, обнаруживать объекты из заданного набора (изображения-маркеры).
- Для каждого обнаруженного объекта выдавать:
  - координаты (3D‑позиция в системе координат робота или глобальной);
  - идентификатор объекта (поскольку объекты представляют собой разные изображения).

#### 4.2. Планирование пути и активное исследование

- Глобальное планирование: построение маршрута к целевому объекту с учётом известных препятствий (данные карты, формируемой в процессе движения).
- Локальное планирование / управление: генерация целевых скоростей (линейная/угловая) для базовой RL‑политики ходьбы, обеспечивающих следование маршруту и обход препятствий.
- Активное исследование: если объект не виден из текущей позиции, робот должен самостоятельно перемещаться по помещению для его поиска. 
- Обработка последовательности: робот должен посещать объекты в строго заданном порядке. После подтверждения достижения текущего объекта переходить к поиску следующего.

### 5. Критерии оценивания решений в первом этапе финала (работа в симуляторе)

1. **Обнаружение объектов** – команда получает очки за каждый объект, который был корректно обнаружен (распознан на изображении с камеры) в процессе выполнения задания. Баллы по данному критерию считаются пор формуле ```С_о = 1*n_o```, где  n_o - количество обнаруженных объектов.
2. **Достижение объектов** – команда получает очки за посещение каждого объекта; объект считается посещённым, если робот на 2 секунды остановился (не давалась команда на изменение положения центра робота) в пределах <= 1.5 метров от центра куба с изображением целевого объекта, и только при соблюдении заданной последовательности достижения целей. Достижение объекта вне очередности не начисляет очков за его посещение. Баллы по данному критерию считаются пор формуле ```С_d = 10*n_d```, где  n_d - количество достигнутых объектов, n_d <= n_o.
3. **Время выполнения** – на выполнение задания отводится максимум 5 минут времени - расчет "физического" времени ведется с привязкой с шагам симуляции, чтобы исключить влияние на результат разницы в вычислительных мощностях компьютеров. Коэффициент за время выполнения задания вычисляется по формуле ```K_t = 1+ (300 - t)/100```, где t - это время выполнения задания в секундах. При обнаружении и/или достижении не всех заданных объектов (или нарушении последовательности достижения) время выполнения считается максимальным t = 300.
4. **Баллы за работу решения** будут считаться  по формуле ```C_s = K_t*(С_о + С_d)```. Метрика выстроена в следующей логике:
- при *достижении всех объектов* в правильном порядке большее количество баллов получает команда *с наименьшим суммарным временем* от старта работы до достижения последнего объекта в заданной последовательности, при условии что все эти объекты также были обнаружены; - количество достижений объектов (с учётом заданной последовательности) за отведенное время ценится выше количества обнаруженных объектов;
- при равенстве общего количества баллов ранжирование между командами осуществляется по времени, затраченному на посещение последнего достигнутого объекта.
Модельный расчет:

t    n_o  n_d  C_s

180  5    5    121

240  5    5    88

300  5    3    35

300  5    1    15

300  3    2    23

300  2    1    12


5. За **уровень/сложность** реализованных алгоритмических решений (знание современных методов и подходов) начисляется от 0 до 10 баллов с шагом 5 баллов. Метрика С_m.  
6. За **качество программной реализации** (работа без ошибок, общее качество кода, соблюдение стандартов программирование, использование современных подходов и технологий) начисляется от 0 до 10 баллов с шагом 5 баллов. Метрика С_а.  
7. **Итоговые баллы** считаются как сумма ```С = C_s + С_m + С_а```.  

### 6. Сбор решений первого этапа
Финальным будет считаться коммит, оставленный не позднее 19:00 в ветке (форке) вашей команды. В это время будет зафиксировано состояние проекта и проведено итоговое тестирование в симуляторе.

*В вашем репозитории должен быть указан номер команды - в названии или начале README файла*. Наличие переданного организаторам репозитория (или архива) является ОБЯЗАТЕЛЬНЫМ условием для проведения итоговой оценки.

### 7. Итоговое тестирование решений первого этапа
В процессе финального запуска решений в симуляции, для каждой команды будет проведено по 5 итераций. Из них 3 будут иметь _одинаковый_ для всех участников сид генерации карты, а 2 выбраны случайным образом. Сиды отличаются расположением объектов поиска и препятствий. После каждой итерации будет получен лог-файл - всего 5; на основании них проводится оценка и подсчет итоговых очков. Будьте готовы запускать итоговое решение на своих устройствах!

### 8. Критерии оценивания решений во втором этапе финала (работа на реальном роботе). 
Баллы за работу решения рассчитываются аналогично первому этапу. Вместо баллов за уровень/сложность реализованных алгоритмических решений и качество программной реализации рассчитываются баллы за качество выступления и защиты своего решения в финале - от 0 до 20 баллов с шагом 5 баллов. Общая сумма баллов считается аналогично первому этапу.


### FAQ
1. Если вы по каким-то причинам не делали форк репозитория, тогда 1) необходимо позаботиться, чтобы он был создан до 19:00 и фиксировал ваше итоговое решение, 2) иначе ровно в 19:00 должен быть отправлен архив с проектом в личные сообщения тг любому из менторов.
2. Внесение изменений в решение после отправки финального коммита в 19:00 не допускается. Заблаговременно убедитесь, что ваш проект работает так, как вы задумывали!
3. _Как фиксируется достижение объекта?_ - Обект считается достигнутым, если центр системы координат робота находится внутри заданного радиуса от обнаруженного объекта и не смещается (не вращается) под действием команд управления в течение 2-х секунд.
4. _Как будет проводиться итоговое тестирование?_ - После отправки и фиксирования всех решений в 19:00 на рабочем компьютере каждой команды организаторами будут запущены по 5 итераций с определенным ранее сидом генерации карты и собраны результирующие лог-файлы.
